# 聊天长连接 `reply-stream`：流式数据与 Markdown 拼接、解析说明

本文说明 `sendMessage` 如何调用 `sendChatMessageStreamApi`，SSE 事件如何解析，以及助手回复的正文（Markdown 源码）如何被**增量拼接**、在何种情况下被**终态覆盖**，最后在界面中如何被 **markdown-it** 解析渲染。

---

## 1. 调用关系

| 层级 | 文件与符号 |
|------|------------|
| 页面入口 | `src/views/home/index.vue` → `sendMessage()` |
| 流式 API | `src/api/chat.ts` → `sendChatMessageStreamApi(sessionUuid, payload, handlers)` |
| Markdown 渲染 | `src/components/MarkdownContent.vue` |

**HTTP 请求：**

- 方法：`POST`
- 路径：`/chat/sessions/{sessionUuid}/reply-stream`
- Header：`Content-Type: application/json`，`Accept: text/event-stream`；带 Bearer Token。
- Body：`ChatReplyRequest`（`content`、`client_request_id`、`images`、`quote`、`model_name`、`auto_task`、`search_modes` 等与 `sendMessage` 中组装的字段一致）。

---

## 2. `sendChatMessageStreamApi`：SSE 如何解析

1. 使用 `fetch` 获取 `ReadableStream`，`getReader()` 分块读取。
2. 使用 `TextDecoder` 将二进制解码为字符串，追加到内部 **`buffer`**。
3. **`flushBlocks`**：按 SSE 惯例以 **空行 `\n\n`** 切分事件块；未结束的尾部留在 `buffer` 中，流结束时再 `flushBlocks(true)` 处理剩余内容。
4. 每个事件块内解析：
   - `event:` 行 → 事件名（若缺省则按实现中的默认处理）；
   - `data:` 行可有多行，拼接成一段字符串后 **`JSON.parse`** 得到 `data` 对象。
5. **`dispatchEvent`** 按事件名调用对应 handler。

**支持的事件名（与前端 handler 对应）：**

| 事件名 | 回调 |
|--------|------|
| `meta` | `onMeta` |
| `start` | `onStart` |
| `delta` | `onDelta` |
| `done` | `onDone` |
| `saved` | `onSaved` |
| `agent_update` | `onAgentUpdate` |
| `search_log` / `search_result` / `search_status` / `search_focus` | 对应 `onSearch*` |

实现见：`src/api/chat.ts` 中 `dispatchEvent` 与 `flushBlocks`。

---

## 3. `delta` 事件：增量正文字段名 `text`（与 `md` 的说明）

在 `delta` 分支中，前端将 JSON 里的 **`text`** 转为字符串传给 `onDelta`：

```ts
handlers.onDelta?.(String(data?.text || ''), data)
```

- **语义上**：这些片段通常是 **Markdown 源码**的增量，拼接后形成完整的一篇「md 文档」字符串。
- **字段名**：当前实现**只读取 `data.text`**。若后端将增量放在 **`md`** 或其它键名下，现有客户端**不会**自动使用，除非后端与前端约定仍使用 `text`，或在 `chat.ts` 中增加兼容（例如 `data?.text ?? data?.md`）。

---

## 4. `sendMessage`：如何把多段 `delta` 拼成一条完整 Markdown 字符串

助手气泡使用 UI 消息对象上的 **`content`** 字段保存 **整段 Markdown 源码**（纯文本，不是 HTML）。

### 4.1 主路径：`onDelta` → 打字机缓冲 → `pendingAssistant.content`

1. `onDelta(deltaText)` 中 `deltaText` 即上一节的 `String(data?.text || '')`。
2. 调用 **`enqueueTypewriter(textChunk)`**：
   - 经 `sanitizeEvidenceTagText`、`sanitizeFirstStreamChunk`（当前多为透传）处理；
   - 首包时可清空占位提示文案；
   - 将片段追加到 **`typewriterBuffer`**；
   - 用 **`setInterval`** 驱动 **`typewriterTick`**，按批量从缓冲取出字符，**追加**到 **`pendingAssistant.content`**。
3. 效果：服务端多次推送的 `delta` 在客户端被 **顺序拼接** 成一条字符串，并带轻微打字机节奏更新界面。

### 4.2 终态：`onDone` 中的 `done.text`

- 从 **`doneData.text`** 读取终态全文（经 `sanitizeEvidenceTagText`）。
- 若 **`finalDoneText` 非空**：清空打字缓冲、停止定时器，并将 **`pendingAssistant.content = finalDoneText`**，避免与后续残留 delta 重复。

### 4.3 持久化回显：`onSaved` 中的 `assistant_message.content`

- 若存在 **`savedData.assistant_message`**，用 **`msg.content`** 覆盖 `pendingAssistant.content`（同样 sanitize）。
- 并根据 `meta` 设置 `renderAsMarkdown`、`preferMarkdown`、`markdownLabel`（例如 `message_kind === 'search_markdown'` 时展示「知识整合文档」等）。

### 4.4 流结束后的收尾

- 若 **`typewriterBuffer`** 仍有剩余，循环调用 **`typewriterTick()`** 直至耗尽。
- 对 **`pendingAssistant.content`** 再做一次 **`sanitizeEvidenceTagText`**。
- 若仍无正文，可能使用 **`savedAssistantContent`**、或 **`listChatMessagesApi`** 拉取最近一条 assistant 消息等方式兜底（见 `sendMessage` 尾部逻辑）。

---

## 5. 拼接完成后：如何「解析」为页面上的 Markdown

**拼接**发生在字符串层：`pendingAssistant.content` = 完整 Markdown 源码。

**解析与渲染**在组件层完成：

1. 当 **`msg.renderAsMarkdown`** 为真且消息非 `pending` 时，模板使用 **`<MarkdownContent :source="msg.content" />`**（见 `index.vue` 消息气泡区域）。
2. **`MarkdownContent.vue`** 使用 **markdown-it**：
   - `html: false`（禁止原始 HTML，降低 XSS 风险）
   - `linkify: true`、`breaks: true`
   - `md.render(source)` 得到 HTML，通过 **`v-html`** 输出到 `.md-content`。
3. 链接在 renderer 中统一加上 `target="_blank"` 与 `rel="noopener noreferrer"`。

因此：**服务端/流式层提供的是 Markdown 文本；浏览器侧用 markdown-it 将 Markdown 转为 HTML 展示。**

---

## 6. 与检索 / 知识整合相关的 UI 元数据

在 **`onMeta`** 中，若 **`meta.search_dispatch.enabled`** 为真，会设置 **`preferMarkdown`**、**`markdownLabel`**（如「知识整合文档」）、检索进度相关展示等。

在 **`onSaved`** 中，若 **`meta.message_kind`** 为 `search_markdown` / `job_markdown` 或 **`meta.render_markdown`** 为真，会强化 Markdown 展示与标签。

这些不影响「字符串如何拼接」，只影响**标签与样式**。

---

## 7. 与任务笔记中变量 `md` 的区别（易混淆）

在 **`index.vue`** 中，部分流程（例如 `syncCurrentJobNoteIntoChatIfNeeded`、`addChildNoteToChat`）使用局部变量 **`md`** 表示从任务笔记拉取的 **整份 Markdown 文本**，再通过 **`saveChatJobNoteMessageApi`** 的 **`markdown_text`** 等 **REST 接口** 写入会话。

该路径与 **`reply-stream` 长连接**不是同一套协议；长连接增量字段在实现上对应的是 **`delta` JSON 的 `text`**（见第 3 节）。

---

## 8. 小结

| 环节 | 说明 |
|------|------|
| 传输 | `POST reply-stream`，SSE，`event` + `data`（JSON） |
| 增量正文 | `delta` 事件 → 读取 **`data.text`** → `onDelta` |
| 拼接 | `typewriterBuffer` 缓冲后写入 **`pendingAssistant.content`** |
| 覆盖 | `done.text` 或 `saved.assistant_message.content` 可覆盖全文 |
| 渲染 | `MarkdownContent` + **markdown-it** 将 `content` 转为 HTML |

若后端字段与 `text` 不一致，需在 `src/api/chat.ts` 的 `delta` 分发处与后端对齐或做兼容映射。
